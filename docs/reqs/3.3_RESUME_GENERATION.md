# 3.3 Resume Generation System

Parent: [0_APPLICATION_REQUIREMENTS.md](0_APPLICATION_REQUIREMENTS.md) Section 3.3

---

## Overview

The recipe system replaces inline text specs with pointer-based references (`{table, id, column}`) so resumes become lightweight "recipes" that assemble content from the knowledge base at generation time. One recipe = one template = one output. Different formats require separate recipes.

---

## 3.3.1 Recipe Schema & Resolution

### Requirements
- [x] `resume_recipes` table (Migration 006) with name, description, headline, template_id, recipe JSONB, application_id, is_active
- [x] `career_history.career_links` JSONB column for proof/portfolio links per employer
- [x] `resolve_recipe()` function resolves recipe JSON into content_map (placeholder -> text)
- [x] Single value resolution: `{table, id, column}` -> SELECT column FROM table WHERE id
- [x] Array resolution: `{table, ids, column}` -> fetch all, join with ` | `
- [x] Assembly resolution (no column): career_history header, education line, certification line, resume_header
- [x] Literal escape hatch: `{literal: "text"}` for content not yet in knowledge base
- [x] Table whitelist security: only bullets, career_history, skills, summary_variants, education, certifications, resume_header. Unknown tables rejected.

### Schema

```sql
CREATE TABLE resume_recipes (
    id              SERIAL PRIMARY KEY,
    name            VARCHAR(200) NOT NULL,
    description     TEXT,
    headline        TEXT,
    template_id     INTEGER NOT NULL REFERENCES resume_templates(id),
    recipe          JSONB NOT NULL,
    application_id  INTEGER REFERENCES applications(id) ON DELETE SET NULL,
    is_active       BOOLEAN DEFAULT TRUE,
    created_at      TIMESTAMP DEFAULT NOW(),
    updated_at      TIMESTAMP DEFAULT NOW()
);

ALTER TABLE career_history ADD COLUMN career_links JSONB;
```

### Recipe JSON Format

**Single value:** `{"table": "bullets", "id": 26, "column": "text"}`

**Array (joined):** `{"table": "skills", "ids": [12, 45, 67], "column": "name"}`

**Assembly (no column):** `{"table": "career_history", "id": 4}` -> assembles from multiple fields

**Literal:** `{"literal": "H.I.G. Capital Acquisition | Press Release"}`

### Assembly Rules
- `career_history` (no column) -> `employer, location {industry}`
- `education` (no column) -> `degree, field | institution - location`
- `certifications` (no column) -> `name | issuer`
- `resume_header` slot "name" -> `full_name, credentials`
- `resume_header` slot "contact" -> `location (note) . email . phone . linkedin`

### Resolution Flow
1. Fetch recipe by ID from resume_recipes
2. Fetch template via recipe.template_id (placeholder .docx + template_map)
3. `resolve_recipe()` iterates recipe JSON, queries each source row, builds content_map
4. Inject `recipe.headline` as `content["HEADLINE"]`
5. Pass content_map + template to `generate_resume()` (formatting logic unchanged)
6. Output .docx

## Key Design Decisions

1. **One recipe = one template = one output.** Different output formats require separate recipes with separate templates.
2. **Headline on the recipe row, not in recipe JSON.** Most frequently changed per-application field. First-class column for querying and list views.
3. **`{literal}` escape hatch should be minimized.** Every literal is content not in the knowledge base. Over time, migrate to source tables.
4. **Existing resume_versions kept for historical reference.** 127 old entries with inline text specs remain accessible. New resumes use recipes.
5. **Template-filling logic unchanged.** `generate_resume()` function, bold formatting, shape preservation, hyperlinks all stay as-is. Only content_map source changes.
6. **Integration tests use real DB data, not mocks.** Snapshot current output, verify recipe output matches exactly.
