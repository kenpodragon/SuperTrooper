# 11.1 Recipe-Based Resume Generation System

Parent: [11_DB_REQUIREMENTS.md](11_DB_REQUIREMENTS.md) Section 11.8.4

## Overview

The recipe system replaces inline text specs with pointer-based references (`{table, id, column}`) so resumes become lightweight "recipes" that assemble content from the knowledge base at generation time. This enables hundreds of resume variants for near-zero storage, makes every piece of content traceable, and ensures the knowledge base stays the single source of truth.

---

## Architecture

```
resume_recipes (new table)
    |
    |-- template_id --> resume_templates (placeholder .docx + template_map)
    |-- recipe JSONB --> slot-to-source mappings
    |-- headline --> recipe-specific headline text
    |-- application_id --> applications (optional, for tailored resumes)

Recipe JSON maps template slots to knowledge base rows:
    "HIGHLIGHT_1": {"table": "bullets", "id": 101, "column": "text"}
    "KEYWORDS": {"table": "skills", "ids": [12, 45, 67], "column": "name"}
    "JOB_1_HEADER": {"table": "career_history", "id": 4}

Generator:
    1. Load recipe -> resolve all references -> build content_map
    2. Load template -> fill placeholders with content_map
    3. Apply formatting rules from template_map
    4. Output .docx
```

---

## Schema: `resume_recipes`

```sql
CREATE TABLE resume_recipes (
    id              SERIAL PRIMARY KEY,
    name            VARCHAR(200) NOT NULL,    -- "V32 AI Architect - Optum"
    description     TEXT,                     -- "Tailored for Optum Director AI Architecture role"
    headline        TEXT,                     -- Recipe-specific headline (first-class column for querying)
    template_id     INTEGER NOT NULL REFERENCES resume_templates(id),
    recipe          JSONB NOT NULL,           -- Slot -> {table, id, column} mapping
    application_id  INTEGER REFERENCES applications(id) ON DELETE SET NULL,
    is_active       BOOLEAN DEFAULT TRUE,
    created_at      TIMESTAMP DEFAULT NOW(),
    updated_at      TIMESTAMP DEFAULT NOW()
);
```

### Schema change: `career_history.career_links`

```sql
ALTER TABLE career_history ADD COLUMN career_links JSONB;
-- Array of {text, url, description} for proof/portfolio links per employer
```

## Recipe JSON Format

Each slot in the recipe maps to a source table, row ID, and column:

### Single value reference
```json
{
    "SUMMARY": {"table": "summary_variants", "id": 4, "column": "text"},
    "JOB_1_BULLET_1": {"table": "bullets", "id": 26, "column": "text"},
    "JOB_1_INTRO": {"table": "career_history", "id": 4, "column": "intro_text"}
}
```

### Array reference (joined with delimiter)
```json
{
    "KEYWORDS": {"table": "skills", "ids": [12, 45, 67, 89], "column": "name"},
    "EXEC_KEYWORDS": {"table": "skills", "ids": [102, 33, 8], "column": "name"},
    "TECH_KEYWORDS": {"table": "skills", "ids": [55, 71, 14], "column": "name"}
}
```

### Assembly references (no column = assemble from multiple fields)
```json
{
    "JOB_1_HEADER": {"table": "career_history", "id": 4},
    "EDUCATION_1": {"table": "education", "id": 1},
    "CERT_1": {"table": "certifications", "id": 3},
    "HEADER_NAME": {"table": "resume_header", "id": 1},
    "HEADER_CONTACT": {"table": "resume_header", "id": 1}
}
```

Assembly rules:
- `career_history` (no column) -> `employer, location {industry}`
- `career_history` (additional exp) -> `title | employer | dates`
- `education` (no column) -> `degree, field | institution, location`
- `certifications` (no column) -> `name | issuer`
- `resume_header` -> assembles HEADER_NAME and HEADER_CONTACT with bullet separators

### Literal text (escape hatch for unmatched content)
```json
{
    "REF_1_LINK_1": {"literal": "H.I.G. Capital Acquisition & $1B+ Scale | Official Acquisition Press Release"}
}
```

Used when content cannot be traced to a source row (e.g., reference links before career_links is populated). Should be minimized over time.

## Resolution Logic

The `resolve_recipe()` function processes the recipe JSON and returns a `content_map` (placeholder -> text):

| Pattern | Action |
|---------|--------|
| `{table, id, column}` | `SELECT {column} FROM {table} WHERE id = {id}` |
| `{table, ids, column}` | Fetch all rows, join values with ` \| ` |
| `{table: "career_history", id}` (no column) | Assemble header from employer/title/dates/location |
| `{table: "education", id}` (no column) | Assemble from degree/field/institution/location |
| `{table: "certifications", id}` (no column) | Assemble from name/issuer |
| `{table: "resume_header", id}` | Assemble HEADER_NAME and HEADER_CONTACT |
| `{literal: "text"}` | Use text as-is |

**Security**: Table names are whitelisted to: bullets, career_history, skills, summary_variants, education, certifications, resume_header. Unknown tables are rejected.

---

## Generation Flow

```
1. Fetch recipe by ID from resume_recipes
2. Fetch template via recipe.template_id (placeholder .docx + template_map)
3. resolve_recipe() -> iterate recipe JSON, query each source row -> content_map
4. Inject recipe.headline as content["HEADLINE"]
5. Pass content_map + template to existing generate_resume() function
6. Formatting logic (bold_label, shapes, hyperlinks) is unchanged
7. Output .docx
```

The existing `generate_resume()` template-filling function is reused as-is. Only the content_map construction changes (recipe-based vs spec-based).

---

## Key Design Decisions

1. **One recipe = one template = one output.** Different output formats (e.g., ATS plain text) require separate recipes with separate templates.

2. **Headline lives on the recipe row, not in recipe JSON.** It's the most frequently changed per-application field and deserves first-class column status for querying.

3. **`{literal}` escape hatch exists but should be minimized.** Every literal is a piece of content not in the knowledge base. Over time, content should be migrated to source tables.

4. **Existing resume_versions table kept for historical reference.** The 127 old entries (with inline text specs and document blobs) remain accessible but new resumes use recipes.

5. **Template-filling logic is unchanged.** The `generate_resume()` function, bold formatting, shape preservation, hyperlinks... all stay exactly as they are. Only the content_map source changes.

6. **Integration tests use real DB data, not mocks.** Snapshot current output text, verify recipe-based output matches exactly.
